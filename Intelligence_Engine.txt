Euda Activity Intelligence Engine
Architecture Specification v1.0

Executive Vision
This is not a web scraper. This is an autonomous intelligence network that aims to know more about real-world activities in a region than any single human, organization, or existing platform possibly could.
The system operates on a core principle: no activity should exist in the physical world without Euda knowing about it. Every trail, every weekly pickup game, every community yoga class, every gear rental shop, every cabin booking—all organized, validated, and ready to assemble into flourishing experiences.

System Philosophy
Why Multi-Layered Architecture?
Single-layer scrapers fail because they conflate responsibilities:
Discovery ≠ Extraction ≠ Validation ≠ Enrichment
When one component tries to do everything, it makes assumptions. Assumptions create errors. Errors compound into garbage data that destroys user trust.
Each layer in this system has ONE job. Layers communicate through well-defined contracts. If a layer can't complete its job with confidence, it escalates—never guesses.
The Confidence Threshold Principle
Every piece of data carries a confidence score (0.0 - 1.0). Data below the threshold doesn't get served to users—it gets routed to resolution. The system would rather show nothing than show wrong information.
CONFIDENCE THRESHOLDS
├── 0.95+ → Serve to users immediately
├── 0.80-0.94 → Serve with soft disclaimer ("hours may vary")
├── 0.50-0.79 → Queue for automated verification
├── 0.20-0.49 → Queue for human-assisted resolution
└── <0.20 → Discard or flag for manual review

Layer 1: Discovery Layer
Purpose
Find and catalog every possible source of activity information for a given region. Does NOT extract data—only identifies WHERE data lives.
Responsibilities
Identify official sources (government, parks, recreation departments)
Identify community sources (Facebook groups, local forums, newsletters)
Identify commercial sources (tour operators, rental shops, event venues)
Identify aggregator sources (AllTrails, Eventbrite, Meetup)
Identify social signals (Instagram location tags, local hashtags)
Maintain source registry with freshness schedules
Score source reliability based on historical accuracy
Source Categories
SOURCE TAXONOMY
│
├── OFFICIAL SOURCES (highest trust)
│   ├── Federal: NPS, USFS, BLM trail databases
│   ├── State: State park systems, DNR
│   ├── County: Parks & recreation departments
│   ├── Municipal: City recreation calendars
│   └── Universities: Campus events, club sports
│
├── AGGREGATOR SOURCES (medium trust, high volume)
│   ├── AllTrails, Hiking Project, MTB Project
│   ├── Eventbrite, Meetup, Facebook Events
│   ├── Yelp, Google Places, TripAdvisor
│   └── Local news event calendars
│
├── COMMERCIAL SOURCES (medium trust)
│   ├── Tour operators & guides
│   ├── Gear rental shops
│   ├── Lodging providers (cabins, campgrounds)
│   ├── Restaurants with event spaces
│   └── Gyms, studios, recreation centers
│
├── COMMUNITY SOURCES (variable trust, high signal)
│   ├── Facebook community groups
│   ├── Reddit local subreddits
│   ├── Nextdoor neighborhoods
│   ├── Discord servers
│   └── Email newsletters (local bloggers, tourism boards)
│
└── SOCIAL SIGNALS (low trust, trend detection)
    ├── Instagram location tags & hashtags
    ├── TikTok local content
    └── Twitter/X local chatter

Discovery Mechanisms
Seed Expansion: Start with known sources, crawl for links to related sources
Search Engine Mining: Programmatic queries for "[region] hiking trails", "[town] events calendar", etc.
Social Listening: Monitor for mentions of activities/events to trace back to sources
User Reports: Euda users can submit sources they know about
Competitive Analysis: What sources do other apps use? (reverse engineer their data)
Anti-Patterns to Avoid
❌ Assuming a source exists because it "should" (validate everything)
❌ Over-relying on aggregators (they have gaps and biases)
❌ Ignoring small/niche sources (often highest signal-to-noise)
❌ Static source lists (sources appear, disappear, change URLs constantly)

Purpose

Handle diverse formats (HTML, JSON APIs, PDFs, images)
Normalize data into unified activity schema
Preserve provenance (where did each field come from?)
Handle rate limiting, auth, anti-bot measures gracefully
Log extraction confidence per field

For each field, rate your confidence (0.0-1.0).

Source content:
{page_content}

{schema}

Anti-Patterns to Avoid
❌ Inferring missing fields (that's Enrichment's job)
❌ Trusting parsed dates blindly (timezones, year ambiguity are common)
❌ Ignoring rate limits (gets sources blocked)
❌ Throwing away raw text (needed for debugging and reprocessing)

Layer 3: Validation Layer
Purpose
Verify that extracted data is accurate, current, and internally consistent. Flag discrepancies for resolution.
Responsibilities
Cross-reference claims against multiple sources
Verify temporal validity (is this event in the future? is this trail open?)
Check geographic consistency (do coords match address?)
Detect duplicates and merge records
Identify stale data that needs refresh
Flag logical impossibilities
Validation Checks
VALIDATION PIPELINE
│
├── TEMPORAL VALIDATION
│   ├── Is start_date in the future? (for events)
│   ├── Does end_date come after start_date?
│   ├── Is the day-of-week consistent with the date?
│   ├── Are operating hours plausible?
│   └── Has this event already passed? (stale detection)
│
├── GEOGRAPHIC VALIDATION
│   ├── Do coordinates fall within stated region?
│   ├── Does address geocode to stated coordinates (±500m)?
│   ├── Is the location physically accessible?
│   └── Does driving time from nearby cities match claims?
│
├── CROSS-SOURCE VALIDATION
│   ├── Do other sources confirm this activity exists?
│   ├── Do details match across sources?
│   ├── Is the organizer verified to be legitimate?
│   └── Are contact details consistent?
│
├── SEMANTIC VALIDATION
│   ├── Does description match activity type?
│   ├── Are requirements plausible for activity?
│   ├── Is pricing in reasonable range for category?
│   └── Are capacity limits realistic?
│
└── FRESHNESS VALIDATION
    ├── When was source last updated?
    ├── When did we last verify this data?
    ├── Does the activity show signs of being outdated?
    └── Are there seasonal considerations?

Anti-Patterns to Avoid
❌ Assuming single-source data is invalid (some activities ARE only listed once)
❌ Over-aggressive duplicate merging (similar events can be distinct)
❌ Ignoring seasonal patterns (ice fishing exists Dec-Mar, not July)
❌ Validating against stale reference data

Layer 4: Resolution Layer
Purpose
Autonomously fill gaps and resolve uncertainties through active outreach. This is the layer that makes Euda's data categorically better than competitors.
Responsibilities
Identify what information is missing or uncertain
Determine appropriate resolution strategy
Execute automated outreach (email, SMS, phone)
Parse responses and update activity records
Escalate to human when automation fails
Learn from resolution outcomes to improve future extractions
Resolution Strategies
RESOLUTION STRATEGY MATRIX
│
├── MISSING HOURS/SCHEDULE
│   ├── Primary: Check Google Places API
│   ├── Secondary: Email business with template
│   └── Tertiary: Call during business hours (AI voice)
│
├── MISSING COORDINATES
│   ├── Primary: Geocode address via multiple APIs
│   ├── Secondary: Search for location on maps
│   └── Tertiary: Ask organizer for directions
│
├── UNCONFIRMED EVENT
│   ├── Primary: Check organizer's social media
│   ├── Secondary: Email organizer for confirmation
│   └── Tertiary: Call organizer
│
├── MISSING REQUIREMENTS (gear, skill level)
│   ├── Primary: Infer from activity type + conditions
│   ├── Secondary: Check similar activities
│   └── Tertiary: Email organizer
│
├── MISSING PRICING
│   ├── Primary: Check booking page
│   ├── Secondary: Email for rate sheet
│   └── Tertiary: Mark as "contact for pricing"
│
├── CONFLICTING INFORMATION
│   ├── Primary: Prefer most recent source
│   ├── Secondary: Email to confirm current info
│   └── Tertiary: Mark all versions with uncertainty
│
└── MISSING CONTACT INFO
    ├── Primary: Search organization website
    ├── Secondary: Check social profiles
    └── Tertiary: Use domain WHOIS (businesses only)

Human Escalation
When automation fails:
Queue for human review with full context
Human can: manually research, make direct calls, mark as unresolvable
Learnings from human resolution feed back into automation rules
Anti-Patterns to Avoid
❌ Spamming businesses with repeated outreach (rate limit per recipient)
❌ Calling at inappropriate times (respect business hours, timezone)
❌ Overly formal/robotic outreach (sounds like spam)
❌ Not tracking outreach history (leads to duplicate contacts)
❌ Assuming no response = invalid (they might just be busy)

Layer 5: Enrichment Layer
Purpose
Add context and metadata that transforms raw activity data into actionable intelligence. This is where Euda's flourishing philosophy gets embedded.
Responsibilities
Calculate flourishing dimension scores
Identify prerequisite needs (gear, skills, transportation)
Link to supporting resources (rentals, guides, lodging)
Generate accessibility assessments
Compute difficulty/intensity ratings
Add weather sensitivity flags
Create optimal timing recommendations
Flourishing Dimension Scoring
Every activity gets scored on Euda's five dimensions:
class FlourishingScorer:
    """Score activities on Euda's flourishing dimensions."""
    
    DIMENSIONS = {
        "agency": "Does this give the user autonomy and choice?",
        "growth": "Does this challenge skills or provide learning?",
        "connection": "Does this involve meaningful human interaction?",
        "meaning": "Does this contribute to purpose or community?",
        "engagement": "Does this enable flow states and presence?"
    }
    
    def score_activity(self, activity: Activity) -> FlourishingProfile:
        prompt = f"""
        Activity: {activity.name}
        Type: {activity.type}
        Description: {activity.description}
        Group size: {activity.capacity}
        
        Score this activity 0-10 on each flourishing dimension:
        
        1. AGENCY: How much autonomy/choice does participant have?
           (solo exploration = high, rigid group tour = low)
        
        2. GROWTH: How much skill challenge or learning?
           (new skill workshop = high, passive spectating = low)
        
        3. CONNECTION: Quality of human interaction?
           (deep small group = high, anonymous crowd = low)
        
        4. MEANING: Contribution to purpose/community?
           (volunteer work = high, pure entertainment = low)
        
        5. ENGAGEMENT: Potential for flow state/presence?
           (active participation = high, passive consumption = low)
        
        Output as JSON with scores and brief justifications.
        """
        
        return llm.structured_output(prompt, FlourishingProfile)

Layer 6: Planning Layer
Purpose
Assemble individual activities into coherent, optimized experiences. This is where "take a hike" becomes "your perfect weekend in the Adirondacks."
Responsibilities
Generate multi-activity itineraries
Optimize for logistics (distance, timing, energy)
Balance flourishing dimensions across a trip
Handle booking coordination
Create contingency plans (weather, availability)
Personalize based on user flourishing profile
Trip Assembly Engine
class TripPlanner:
    """Assemble activities into optimized multi-day experiences."""
    
    def plan_trip(
        self,
        user: UserProfile,
        constraints: TripConstraints,
        preferences: TripPreferences
    ) -> Trip:
        """
        Generate an optimal trip plan.
        
        Args:
            user: User's flourishing profile, skill levels, gear owned
            constraints: Dates, budget, party size, base location
            preferences: Activity types, intensity, must-dos
            
        Returns:
            Complete trip with daily itineraries, bookings, packing list
        """
        
        # 1. Candidate activity retrieval
        candidates = self.get_candidates(
            region=constraints.region,
            dates=constraints.dates,
            types=preferences.activity_types
        )
        
        # 2. Filter by hard constraints
        viable = self.filter_viable(
            candidates,
            skill_level=user.skill_levels,
            budget=constraints.budget,
            party_size=constraints.party_size
        )
        
        # 3. Score by flourishing fit
        scored = self.score_flourishing_fit(
            viable,
            user_profile=user.flourishing_profile,
            growth_edges=user.growth_edges
        )
        
        # 4. Optimize daily schedules
        schedule = self.optimize_schedule(
            scored,
            dates=constraints.dates,
            base_location=constraints.lodging or constraints.home,
            energy_model=user.energy_model
        )
        
        # 5. Add logistics
        trip = self.add_logistics(
            schedule,
            gear_owned=user.gear_inventory,
            transportation=user.transportation
        )
        
        # 6. Generate contingencies
        trip.contingencies = self.generate_contingencies(
            trip,
            weather_forecast=self.get_forecast(constraints.region, constraints.dates)
        )
        
        return trip

Optimization Algorithms
Activity Selection: Multi-objective optimization balancing:


Flourishing fit (match user's growth edges)
Logistical efficiency (minimize driving)
Energy management (don't stack exhausting activities)
Variety (different activity types, environments)
Temporal Scheduling: Constraint satisfaction:


Operating hours
Travel times between locations
Energy curves (demanding activities when fresh)
Meal timing
Weather windows
Budget Allocation: Knapsack-style optimization:


Prioritize must-do experiences
Find value alternatives for nice-to-haves
Account for hidden costs (parking, tips, gear)
Anti-Patterns to Avoid
❌ Over-scheduling (leave breathing room)
❌ Ignoring travel time (activities too far apart)
❌ All-high-intensity (leads to burnout and injury)
❌ Generic recommendations (must reflect user's specific profile)
❌ Unbookable plans (verify availability before suggesting)

Error Handling Strategy
class OrchestrationErrorHandler:
    """Handle errors across the pipeline."""
    
    ERROR_POLICIES = {
        "extraction_failed": {
            "retry_count": 3,
            "retry_delay_minutes": [5, 30, 120],
            "fallback": "mark_source_unhealthy",
            "alert_after_retries": True
        },
        "validation_timeout": {
            "retry_count": 2,
            "retry_delay_minutes": [10, 60],
            "fallback": "queue_for_manual_review",
            "alert_after_retries": False
        },
        "resolution_no_response": {
            "retry_count": 2,
            "retry_delay_days": [3, 7],
            "fallback": "mark_low_confidence",
            "alert_after_retries": False
        },
        "enrichment_api_failure": {
            "retry_count": 5,
            "retry_delay_minutes": [1, 5, 15, 60, 240],
            "fallback": "enrich_partial",
            "alert_after_retries": True
        }
    }
    
    def handle_error(self, error: PipelineError):
        policy = self.ERROR_POLICIES.get(error.type)
        
        if error.attempt_count < policy["retry_count"]:
            self.schedule_retry(error, policy)
        else:
            self.execute_fallback(error, policy)
            if policy["alert_after_retries"]:
                self.alert_ops_team(error)

Success Metrics
Coverage Metrics
Activity Coverage: % of real-world activities in region that are in database
Source Coverage: # of unique sources per region
Category Coverage: Activities across all types (hikes, events, classes, etc.)
Quality Metrics
Confidence Distribution: % of activities at each confidence tier
Accuracy Rate: % of served data confirmed accurate (sample audit)
Freshness: Average age of data, % stale
Resolution Metrics
Resolution Rate: % of gaps successfully filled
Response Rate: % of outreach attempts that get responses
Time to Resolution: Median time from gap identified to resolved
Planning Metrics
Trip Completion Rate: % of generated trips that users actually do
Booking Conversion: % of suggested bookings completed
User Satisfaction: Ratings on trip quality
Operational Metrics
Pipeline Throughput: Activities processed per day
Error Rate: % of jobs that fail
System Uptime: % availability

Open Questions for V1.1
Legal/Compliance: What are the terms of service implications for scraping various sources? Need legal review.


User-Generated Data: How do we incorporate activities that users report? Trust scoring?


Real-Time Events: How do we handle truly real-time info (pickup games forming now, flash events)?


Booking Integration: Do we book on behalf of users or just link out? Liability implications?


Multi-Language: If expanding beyond English-speaking regions, how does extraction change?


Offline Resilience: How does the system behave when external APIs are down?


GDPR/Privacy: If we're storing contact info for organizers, what are the compliance requirements?


Voice AI Disclosure: What's required when AI calls businesses? "This call may be recorded"?
